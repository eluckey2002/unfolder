# Session 0031 — v3.5 spike: region re-unfold algorithm

**Work type:** spike (first session of the v3.5 phase per v4 design spec § 3).
**Branch:** `spike/0031-region-reunfold`.
**Land via:** worktree → PR → CI green → squash-merge per ADR 0006.

## Goal

Design the region re-unfold algorithm and validate it via a worked example on `deer.obj`. Output is a written algorithm specification plus a worked-example walkthrough committed under `docs/spikes/2026-05-17-v3.5-region-reunfold/`, sufficient to ground the v3.5 promote-to-core session and flag any required revisions to v4 design spec § 7.

No production code in `src/core/` changes. No perf measurement this session — perf cost model is the v3.5 second-half deliverable per v4 spec § 4.

### Exit criteria

A verdict of one of:

- **ALGORITHM-READY** — algorithm spec is unambiguous, worked example succeeds, promote-to-core proceeds in session 0032.
- **NEEDS-REVISION** — specific underspecifications or worked-example failures named; second spike session 0032 closes them.
- **INFEASIBLE** — the algorithm cannot be made deterministic + correct + tractable within the v4 design spec's contract. Fallback per § 3 (manual region-pin + full-pipeline-re-run with loading spinner) becomes v4.3's ship plan; ADR draft for that posture is the 0032 deliverable instead.

### Time-box

Two sessions for the spike half of v3.5 (0031 + 0032). If 0031 returns NEEDS-REVISION, the strategist decides whether 0032 closes the spike or further sessions are warranted. If still inconclusive after 0032, declare INFEASIBLE.

## Tasks

The implementer drafts the atomic plan in plan mode before any artifact, per CLAUDE.md § 1. Spike sessions substitute **research → write → review → revise → commit** for the standard TDD shape — no production tests this session. Plan gates are report-form (no "X equals N" predictions).

Sketch of the task arc; the implementer refines decomposition in plan mode.

1. **Read v4 spec § 7 in full and enumerate underspecifications.** Walk the pseudocode block. List every step that is sketched rather than specified — known candidates from a strategist read:
   - `pickRegistrationAnchor(region, layout)` — behavior on a region with no fold-edge boundary (region detaches entirely); behavior on regions with multiple candidate fold edges.
   - `resolveObstacle(layout, obstacleSet)` — what transformations it tries (rotation about anchor? translation? both?), in what order, with what termination criterion (angular step size? overlap-area threshold?).
   - `layout.replaceRegion(region, globalLayout)` — what happens to pre-existing layout pieces that intersected the region but were not in it; how page assignment is preserved or recomputed.
   - `edgeOrderHash` — which edges contribute (region-only? region + boundary? entire mesh?); how the hash is computed (canonical edge ID list, hashed how).
   - FAIL handling — does the splice mark the fix failed silently, or does the engine surface it as a fix family the user sees ("could not apply automatically" per v4 spec § 7)?
   - The relationship between `seed: u32` and the spanning-tree / cut-removal RNG paths in `src/core/`. Current code may not be seedable end-to-end — surface if so.

2. **Algorithm draft.** Write the algorithm to `docs/spikes/2026-05-17-v3.5-region-reunfold/algorithm.md`. Structure: input/output contract, preconditions, step-by-step procedure with rationale and edge-case handling, failure modes (each step that can fail and what the algorithm does), determinism contract (precise definition of `edgeOrderHash` and `seed` binding), API sketch (function signature in TypeScript, types referenced from `src/core/` — do not commit the types themselves to `src/`). Function names from v4 spec § 7's pseudocode (`pickRegistrationAnchor`, `resolveObstacle`, `layout.replaceRegion`, etc.) are illustrative — the spike may rename for clarity, but the behavior contract is binding.

3. **Pick a target region on `deer.obj`.** Per-piece counts are in `docs/baseline-pipeline.md` (post-0028 baseline; the canonical source). If a fresh render is wanted, `pnpm baseline` regenerates it. Recommended target depends on Task 4's trace path: if hand-tracing, pick a region small enough to fit on paper (a handful of faces); if scripted, pick the largest internal piece (~50 faces per v4 spec § 4's perf-target benchmark) or a piece adjacent to a warn-classified region so the example exercises a realistic fix scenario. Document the choice, the trace path, and the actual face count.

4. **Worked-example trace.** Either by hand (per Task 3's sizing call) or with throwaway scratch code at `scripts/probe-0031-spike.ts` — transient, NOT committed. The probe pattern: a `vite-node` script that imports `runPipeline` from `src/core/pipeline.ts`, parses an OBJ from `test/corpus/`, writes scratch output (logs, intermediate SVGs) to `os.tmpdir()`, and is deleted with `rm scripts/probe-0031-spike.ts` before commit. Same shape as `probe-v3-sweep.ts` from session 0029 (described in `docs/sessions/0029-v3-close.md`; not on disk because it was deleted on cleanup). For each algorithm step on the chosen region, document the actual output: obstacle-set composition (face count, polygon count), registration-anchor choice and the pinned endpoint coordinates, local pipeline run results (spanning-tree shape, flatten output, recut output), obstacle-resolution attempts (each transformation tried and its overlap result), splice result, final `detectOverlaps` pass/fail.

5. **Surface and resolve ambiguities.** Each underspecification from Task 1 receives a written resolution in `algorithm.md`. Any new ambiguity exposed by the worked example is added and resolved. If a resolution requires changing v4 spec § 7 (rather than just filling in unspecified detail), flag it in `findings.md` — the spec change happens in the 0032 promote-to-core session, not here.

6. **Verdict.** Write `docs/spikes/2026-05-17-v3.5-region-reunfold/findings.md`. Three sections: verdict (one of the three exit-criteria values), evidence (what the worked example actually showed), next step (0032 scope concretely — promote-to-core / second spike session / fallback ADR).

7. **Session log.** Standard `docs/sessions/0031-v3.5-region-reunfold-spike.md`. Document the verdict, link the spike directory, name any v4 spec § 7 revisions that surfaced (those propagate back to the spec in 0032, not here), report scratch-code cleanup.

## Specs

**The algorithm spec is the deliverable.** Behavior — not call signatures — but precise enough that the 0032 promote-to-core session can implement directly from it without re-deriving design intent.

**Determinism contract.** `algorithm.md` defines `edgeOrderHash` precisely enough that two independent implementations would produce the same hash on the same region. The v4.3 determinism property test depends on this; the 0030 store-shape commitment locks the field type in.

**Worked-example completeness.** A reader who has not seen the algorithm should be able to follow the worked example end-to-end. Numbers (face counts, polygon counts, transformations applied, overlap-check results) are reported in report-form — actual values from the trace, not predicted ahead.

**Spike posture.** Throwaway scratch code allowed in `scripts/probe-0031-spike.ts`; cleaned up before commit. The spike's committed output is the docs under `docs/spikes/2026-05-17-v3.5-region-reunfold/`. No production code in `src/core/` changes. No baseline regen.

**Visual gate.** Not applicable to algorithm-design output. If the worked example renders intermediate-state SVGs to make the trace legible, those live as static images in the spike directory and are committed only if load-bearing for the worked example. CLAUDE.md § 1 names `src/core/emit-svg.ts`, `src/core/paginate.ts`, `src/core/foldability.ts` as the visual-gate triggers; this session touches none.

**Sub-agent reliability calibration** (v3-retrospective Decision 1, the v4 pilot). Interpretive sub-agent tasks (review the algorithm draft for gaps, score the worked example for honesty, check the algorithm against v4 spec § 7) are trusted by default. Enumerative claims the worked example depends on (face counts, edge counts, overlap-check counts) are cross-checked with deterministic tooling (grep on the OBJ, scratch-code printf, manual count) before the worked example commits.

**Report-form gates** (v3-retrospective Decision 3 — CLAUDE.md § 1). Plan-mode gates of the form "the worked example will find N ambiguities" or "deer has K pieces" are predicted-value smells. Use report-form: "report the number of ambiguities surfaced," "report the actual region face count chosen."

**Out of scope (do not implement this session):**

- Perf measurement / cost model — deferred to 0032 or the promote-to-core session per v4 spec § 4. Algorithm correctness first; perf second.
- Production code in `src/core/` — promote-to-core is 0032's scope.
- ADR — v3.5 gets an ADR at promote-to-core if ALGORITHM-READY, or at INFEASIBLE if that verdict ships. Spike output is markdown findings, not ADR.
- v4 spec § 7 edits — flagged here, applied in 0032. Keeps the spec stable across spike and promotion.
