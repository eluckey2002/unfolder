## Goal

Replace `src/core/tabs.ts`'s "lower mesh-face index gets the tab" rule with a score-driven choice between the two candidate sides of each cut edge. Score by edge length minus a hard penalty when the candidate tab polygon would overlap the originating piece's own interior. Algorithmic no-op on convex (single-face piece) corpus models; reshapes tab placement on concave (multi-face piece) corpus models — `croissant.obj`, `deer.obj`, `ginger-bread.obj`.

## Specs

- `scoreTabPlacement({ edgeLengthMm, tabOverlapsOwnPieceInterior })` — exported. Score is `W_LENGTH * edgeLengthMm` minus a hard penalty when `tabOverlapsOwnPieceInterior` is true. Starting constants: `W_LENGTH=1.0`, `W_OVERLAP=1000` — **tunable**; the penalty must dominate any reasonable edge-length term so an overlap-free candidate always wins regardless of edge length. If the weights shift during implementation, record the final values in the decisions-log entry from Task 26.6.
- `tabOverlapsOwnPieceInterior(tabPoly, pieceFaces, originatingFaceIdx)` — exported. Tests the candidate tab polygon against every other face's positions polygon in the same piece via the `polygonClipping.intersection` pattern from `src/core/overlap.ts`. Mirror the try/catch defense from that file (treat exceptions as non-overlap, same numerical reasoning).
- `buildRenderablePieces` becomes two-pass: pre-pass decides per-cut globally; main pass looks up the winning side. Tie-break to `adj.faceA` (preserves existing convex-model fixture expectations).
- `scripts/baseline-pipeline.ts` gains a `tab overlap (own)` column. After Task 26.3 lands, expect 0 on every row across the 11-model corpus — if any row > 0 the algorithm has a bug, go back to Task 26.3 Step 3.
