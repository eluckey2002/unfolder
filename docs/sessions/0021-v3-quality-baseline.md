# Session 0021 — v3 quality baseline

## What was attempted

Establish v3's quality-measurement baseline. Define the metric set
that gives v3's improvement vectors (optimized recut, smart tab
placement, packing) something concrete to move; instrument
`scripts/baseline-pipeline.ts` with the new metrics; capture the
frozen "v3 starts here" snapshot in `docs/baseline-v3.md`; extend the
CI baseline guard to cover both the live and frozen files. First
session run under the PR flow stood up by 0020 (ADR 0006).

## What shipped

- **`scripts/baseline-pipeline.ts` extended** with three quality
  columns — `cut length (mm)`, `tabs`, `paper efficiency`. Each is
  computed from data the pipeline already produces, with no new
  pipeline stage:
  - **Cut length** sums the 2D mm-length of every `cut`-kind edge
    across the renderable pieces, scaled by paginate's uniform `s`.
    Because cuts separate pieces and each side appears in its own
    piece's outline, this counts both sides — which matches actual
    builder effort, since pieces are placed separately on the page
    and the same edge is cut once per piece outline.
  - **Tab count** is `recutResult.cuts.length` — under the
    lower-face-index tab rule in `src/core/tabs.ts`, every cut
    yields exactly one tab.
  - **Paper efficiency** is the sum of 2D face-triangle areas (from
    `Piece.layout.faces[].positions`) multiplied by `s²` (post-scale
    mm²), divided by `pages × printable_per_page`, reported to one
    decimal place as a percentage. The scale `s` is computed
    locally by replicating paginate's most-constrained-piece logic
    so the script does not depend on paginate exporting `s`.
- **`docs/baseline-pipeline.md` regenerated** with the ten-column
  table and an extended summary line that reports total cut length,
  total tabs, and average paper efficiency. Second consecutive run
  is byte-identical modulo the date stamp.
- **`docs/baseline-v3.md` created** as the frozen "v3 starts here"
  snapshot — preface naming the five metrics and their improvement
  vectors, then a horizontal rule, then the table and summary copied
  verbatim from `docs/baseline-pipeline.md` (title line, blank, and
  `Generated by …` header stripped per spec). One-time capture,
  never auto-regenerated.
- **`.github/workflows/ci.yml` Baseline guard extended** to check
  both files. The live baseline keeps its `-I '^Generated by'`
  ignore for the volatile date line; the frozen baseline has no
  volatile lines so no ignore is needed. The single
  `baseline-change` label remains the escape hatch and covers both
  files. On clean diff, success; on dirty diff without label, the
  step prints a `::error::` line for each dirty file and exits 1;
  on `push: main` the label is absent so the post-merge backstop
  still asserts both clean.
- **Decisions-log entry appended** recording the v3 quality metric
  set verbatim per the prompt.
- **`docs/roadmap.md` advanced** — "Where we are now" now reads last
  completed 0021 / next planned 0022; the v3 session plan marks 0020
  ✅ (it was still ⏭ from its own session), 0021 ✅ with a one-line
  shipped summary, 0022 ⏭.

## Decisions made or deferred

- **Five-metric set, foldability deferred.**
  `[surfaced-and-proceeded]` — captured in `docs/decisions-log.md`.
  The four numeric metrics (piece count, cut length, tabs, paper
  efficiency) plus page count cover the four v3 improvement vectors
  with measurable signal each. Foldability would be a fifth quality
  axis but no clean measurable definition is in hand; it is
  explicitly deferred rather than approximated with a noisy proxy.
- **Cut length sums both sides.** `[flowed-silently]` — the spec
  said "walk each piece's edges, summing the 2D mm-length of every
  edge classified as a cut." This naturally double-counts cuts
  shared between two pieces, but that matches actual builder effort
  since pieces are placed separately on the page and the same edge
  is cut once per piece outline. Documented above for future
  reference if it ever surfaces as a question.
- **Scale `s` is recomputed in the harness, not exported from
  paginate.** `[flowed-silently]` — keeping paginate's API surface
  unchanged is cheaper than threading a new return value, and the
  scale computation is a six-line copy of paginate's
  most-constrained-piece logic. If a second consumer ever needs it,
  promoting it to a paginate export is a one-line refactor.

## Concerns, uncertainties, questions

- **CI baseline guard tested only locally.** The YAML parses
  (verified with `python3 -m yaml`) and the diff/label logic is a
  straight extension of the 0020 pattern. The first PR — this one —
  is the first live test of both the extended guard and the
  squash-merge path. Confidence: high on syntax; medium on whether
  the `baseline-change` label exists yet on the GitHub repo (0020's
  handoff flagged this for Evan's GitHub-side setup).
- **Cut-length double-counting interpretation.** Settled above as
  matching builder effort, but worth surfacing in case the
  strategist had a single-count semantics in mind. The numbers would
  halve under the other interpretation; the v3 trajectory (relative
  change across sessions) is unaffected either way.
- **Paper efficiency on the procedural cube.stl outlier.** cube.stl
  reports 41.0% while cube.obj reports 23.4% — same 12-face shape,
  same pipeline, different scale `s`. The cause is the leading STL
  vs OBJ scale difference in the source corpus (the procedural STL
  cube is dimensioned differently from the procedural OBJ cube),
  which carries through paginate's most-constrained scale. Not a
  bug; it just means STL/OBJ pairs are not directly comparable. The
  v3 improvement vectors operate per-model so this does not affect
  measurement of progress.
