# Critique — Architecture and algorithm risks
Reviewer: independent (no spec authorship; no prior conversation context)
Date: 2026-05-17

## Severity scale
- BLOCKER: spec cannot ship as written; rework required before commit
- HIGH: real risk; spec should be revised but commit possible with caveats
- MEDIUM: worth fixing inline; not gating
- LOW: nit / suggestion

## Issues

### Issue 1 — Region re-unfold's "splice + obstacle" mechanism is hand-waved at exactly the place the algorithm is hard
**Severity:** BLOCKER
**Spec section:** §7 (Region re-unfold mechanism)
**Problem:** The pseudocode treats `pinnedBounds` as a `boundingPolygon` and `positionAroundObstacle(localLayout, pinnedBounds)` as a primitive that exists. Neither is defined. Two concrete failure modes are unresolved: (a) the pinned set is the *complement* of the region — its boundary is generally non-convex, multiply-connected, and possibly has holes the region must fit through; treating it as a single bounding polygon either over-constrains (rejects valid fits in concavities) or under-constrains (admits fits that overlap the actual pinned faces). (b) `localLayout = flatten(localTree)` produces a layout in *the region's* canonical frame — it is not registered against the global frame, so the "splice back" step is doing an implicit rigid alignment that has to share at least one face (or edge) with the pinned set, which means the region's outer faces cannot themselves move, which contradicts the "free to re-unfold" premise. The actual algorithm is closer to a constrained packing problem (place the region's connected components into the global frame avoiding the pinned polygon set, while satisfying the region's own folds and the pinned/region interface folds) — not visible in §7.
**Why this matters:** This is the load-bearing differentiation move for v4.3. If the algorithm is undefined the v3.5 spike will discover it has to invent it, and the spec's "1s budget on 50-face islands" target floats free of any actual design.
**Recommended action:** Expand §7 with an explicit account of (i) what "region" means topologically (face set; what about its boundary edges to the pinned set — fold or cut?), (ii) what data structure represents the pinned set as an overlap obstacle (face polygons, not bounding polygon), (iii) how the region's local layout is registered against the pinned frame (which fold edges anchor it), and (iv) the fallback when no valid placement exists. Promote the v3.5 spike's first deliverable to "produce a written algorithm with worked example on `deer.obj`," not "hit a perf number."

### Issue 2 — "Everything derived from `(mesh + config + pinned + appliedFixes[:N])`" is not true if any fix uses cut-removal
**Severity:** HIGH
**Spec section:** §4 (Why this shape), §7 (Region re-unfold)
**Problem:** ADR 0007 and `src/core/cut-removal.ts` show the v3 default unfolder is iteration-order-dependent (edges sorted by length descending, walked greedily). The result is deterministic given identical input, but it depends on the *full mesh's* edge length ordering — not the region's. Region re-unfold restricted to a face subset will produce a different walk order, and `polygon-clipping.intersection` throws on near-coincident edges treated as conservative overlap rejections (ADR 0007 "5–25% of merges rejected this way"). So: (a) re-applying a fix at index N may not reproduce the layout that was on screen when the user accepted it, because the global edge ordering has changed *or* the polygon-clipping rejection set is sensitive to the obstacle polygon's exact geometry; (b) undo-via-traversal at step N replays fixes 0..N against a layout that may diverge from what the user saw, breaking the "correct by construction" claim. The spec asserts re-derivation; the algorithm is non-monotone in inputs in a way that makes re-derivation a leap.
**Why this matters:** If `currentLayout` at undo step N differs from the layout the user accepted at step N, undo silently corrupts history. This is the trust-collapsing failure mode the spec is otherwise careful about.
**Recommended action:** Either (i) make region re-unfold *capture* the post-condition layout (i.e., store a layout delta in the fix entry, accept the memory cost), or (ii) define a deterministic registration: pin a specific edge order + RNG seed in the fix entry so replay is bit-exact. Add a property test: "applying fixes 0..N from scratch matches the cached layout at step N." Without one of these, drop the "no layout snapshots" claim.

### Issue 3 — The sub-second perf claim ignores non-region-proportional costs
**Severity:** HIGH
**Spec section:** §4 ("region re-unfold must be sub-second"), §11
**Problem:** Current full pipeline on the 200-face corpus is ~1–3s; deer.obj's cut-removal alone is ~72ms per ADR 0007. The §4 inference is "region size → time roughly proportional," so a 50-face region should be far under 1s. Three costs break the proportionality: (a) `detectOverlaps(spliced)` is global, not local — the spec's post-condition assert `detectOverlaps(spliced).isEmpty` runs on the full layout (O(n²) bbox prefilter, then polygon-clipping per candidate pair), and on the 720-face deer the corpus baseline reports ~857 overlaps pre-recut; (b) `positionAroundObstacle(localLayout, pinnedBounds)` is presumably some kind of free-placement search over the obstacle — search cost depends on obstacle complexity (the pinned set), not region size; (c) building the *region's* spanning tree depends on the region's adjacency subgraph, which has to be extracted from the global dual. None of these are linear in region size; (a) is quadratic in *global* face count. The 50-face target is unsourced; deer.obj has 17 pieces from 720 faces post-cut-removal, so a "50-face region" is plausibly a large piece on a complex model, and the global-overlap-check cost will dominate.
**Why this matters:** The spec uses the sub-second budget as the v4.3 viability gate. If the budget is wrong by an order of magnitude on the corpus the v3.5 spike will return "infeasible at this budget" and the v4.3 release plan is invalid.
**Recommended action:** Replace the §4 "roughly proportional" claim with an explicit cost model that names the global checks (overlap detection on splice, post-condition assert) and budgets them separately from region-local work. Make the v3.5 spike measure the global checks as a first deliverable. Consider a *local* overlap check (region faces vs pinned obstacle only) with a separate, optional global post-condition.

### Issue 4 — Fix scoring is gameable: predicates do not cover the costs a fix introduces
**Severity:** HIGH
**Spec section:** §7 (Ranking + "wrong fix" mitigation, REJECT if SCORE < 0)
**Problem:** SCORE counts badge-improvements minus badge-regressions. The badge set (§7 buildability signals) covers piece-level geometry (overlap, page-fit, edge length, angle, aspect, density). It does *not* cover: (a) **assembly difficulty** — splitting an island raises piece count, increases tabs, adds glue work; no badge fires, so split-island looks free; (b) **tab visibility on the spliced boundary** — region re-unfold creates new cut edges at the region boundary, which generate new tabs whose visibility against inside/outside isn't a piece-local predicate the badge re-evaluates on the spliced result; (c) **page count / paper efficiency** — repaginate-with-hint can shrink pieces and increase pages; no badge fires; (d) **piece reorientation cost** — destroys spatial mental model the user built; not a badge. So a "split-island" fix that turns one yellow-aspect-ratio piece into three green pieces scores +1 and ships, even though it doubled the user's cutting and gluing work. Are the badges orthogonal? Mostly yes within their predicates, but they collectively miss the assembly-effort axis Marcus is optimizing for.
**Why this matters:** This is the "wrong fix suggestion" failure mode the findings doc explicitly flagged as trust-destroying. SCORE > 0 is necessary but not sufficient.
**Recommended action:** Either (a) add cost-side terms to SCORE (piece-count delta, tab-count delta, page-count delta, with weights), and surface them in the preview alongside badge deltas; (b) make the per-fix PREVIEW show the cost-side numerics explicitly (Δpieces, Δtabs, Δpages) so the user sees the tradeoff even if SCORE doesn't penalize it. Option (a) is the principled fix; (b) is the minimum.

### Issue 5 — Inside/outside inference's open-edge fallback assumes simply-connected exterior
**Severity:** MEDIUM
**Spec section:** §7 (Inside/outside inference)
**Problem:** "Open-edge mesh: pick three exterior faces → flood-fill the connected exterior region." Flood-fill works only if the exterior face set is connected via face-adjacency. Two corpus-plausible failures: (a) **terrain ruins** can have multiple disjoint exterior regions if the mesh wraps a hollow interior with several open windows — the user has to pick a seed in each region or the unseeded regions stay unclassified; (b) **watertight-but-inverted meshes** (winding reversed by export pipeline — common Blender export glitch) will pass the watertight check and infer the *interior* as outside; the badge "tab on visible face" will then fire on every correctly-placed tab. There is no parity check in the spec (e.g., compare inferred outside-normal direction against geometric centroid → should point away). The "user can always override" path is correct but only kicks in if the user *notices* the wrong inference; the failure mode is silent.
**Why this matters:** Inside/outside drives a load-bearing badge ("tab on visible face") and tab placement. Silently wrong inference produces silently wrong fix suggestions on a class of meshes that aren't rare (Blender exports specifically).
**Recommended action:** (a) Add a sanity check: after watertight-normal inference, verify outward normals point away from mesh centroid for ≥50% of faces; flag a preflight warning if not. (b) For open-edge flood-fill, walk all unclassified faces after the first flood and prompt for additional seeds if any remain. (c) Make inside/outside visualization in 3D more legible than "subtly emphasized in lighting" so silent inversions become spot-able.

### Issue 6 — Persistence design choices in v4.0 will require rework when v4.1 lands save/load
**Severity:** MEDIUM
**Spec section:** §3 (sequencing), §4 (Persistence)
**Problem:** v4.0 ships with the seven-layer state shape (so the store is already designed for save/load) but defers save/load to v4.1. Two specific concerns: (a) `meshHash` is required by the persisted format but the v4.0 mesh-loading path (the absorbed session 0029 file-loader UI) doesn't have to compute one — and the choice of hash (SHA-256 over canonical bytes? over the parsed mesh? over normalized vertex order?) is load-bearing for round-trip correctness and not specified. If v4.0 ships without it and v4.1 picks a different canonicalization than the in-flight session 0029 produced, existing projects break or the spec needs migration logic on day one. (b) IndexedDB schema is unspecified: keyed by mesh content hash means hash-collision-resistant choice matters, and the LRU eviction policy mentioned in §9 implies metadata tables (last-access timestamp, size) the schema needs from day one. (c) `appliedFixes` entries embed `regionSelector` — a face-set reference that depends on mesh face indices being stable across re-attachment. If the OBJ parser ever changes face ordering (a real risk during v3-color-passthrough kind of work), every saved project breaks.
**Why this matters:** v4.0 either ships "no save/load" cleanly (and these decisions are deferred), or it ships the state shape that *implies* save/load and incurs the decisions tacitly. The spec straddles.
**Recommended action:** Either (a) commit the meshHash + IndexedDB schema + face-index stability contract in §4 as part of v4.0 (since the state shape commits to them anyway), or (b) explicitly mark `regionSelector`, `pinnedRegions`, and `appliedFixes` as v4.0 in-memory-only (no persistence-format commitment) and defer their schema to v4.1. The current "designed for save/load, deferred to v4.1" is the worst of both.

### Issue 7 — `src/core/` library boundary risk on the fix-suggestion engine
**Severity:** LOW
**Spec section:** §4 ("No React imports cross into core/"), §7
**Problem:** The fix-suggestion engine in §7 produces ranked `Suggestion` objects with "preview" payloads — the spec describes preview as "side-by-side: current piece + predicted piece + delta." A naive implementation will define a `Suggestion` type that carries human-readable strings ("Split this island along edge 47"), preview thumbnails, drawer-UI hints, etc. That type belongs in `src/app/`, not `src/core/`. The pure-function discipline ADR 0001 establishes is: core produces structured data (face sets, predicate outputs, geometry); app shapes it for the user. The fix-suggestion engine sits exactly on this seam.
**Why this matters:** The seam erodes quietly under feature pressure unless named in the spec.
**Recommended action:** Add a one-line §4 amendment: the fix-suggestion engine in `core/` returns a structured `FixProposal` (action kind + parameters + predicted layout + predicted badge set); the human-readable copy, drawer ordering, and preview rendering live in `src/app/`. Make this a code-review checkpoint.

## Strengths

- **The append-only `appliedFixes` log is the right shape for this problem.** Modulo Issue 2's determinism caveat, the choice to model edits as a fix list (not a layout-state-stack) matches how the v3 pipeline is already structured: pure functions, derived state, no hidden mutation. It is the lowest-overhead store design that gives undo/redo for free, and it composes with save/load naturally.

- **Sequencing v4.0–v4.2 to ship without region re-unfold is the right risk-management move.** The v3.5 spike is the single highest-risk dependency in the spec; isolating it to v4.3 means the first three releases ship on known algorithms (existing pipeline) and the differentiation lands on a foundation the user already trusts. The findings doc's "region re-unfold is C's biggest risk imported wholesale" is honored by the sequencing.

- **The PREDICT → SCORE → PREVIEW → REJECT pipeline (§7) is the right ordering** — predict-before-show is the discipline that prevents the trust-destroying "fix that makes it worse" failure, even with Issue 4's caveats about score coverage. Naming REJECT-if-SCORE<0 as a hard rule is more disciplined than most "suggestion engines" ship with.

- **`src/core/` purity discipline carried forward from ADR 0001 is consistently invoked.** The spec explicitly forbids React imports into core, names the four surfaces as app-level concerns, and treats the pipeline as a library. This is the structural commitment that has paid off through v1–v3 and the spec preserves it.

## Open questions for the spec author

1. **What is the actual data structure for `pinnedBounds`** in region re-unfold? Is it a single polygon (and thus loses concavity), a set of face polygons (and thus an obstacle set), or something else? The answer determines whether `positionAroundObstacle` is a packing solver, a placement heuristic, or just the existing overlap detector.

2. **Is region re-unfold expected to be deterministic given `(region, constraints, pinned)`**, and if so, does that imply either (a) a captured RNG seed in `appliedFixes`, or (b) replaying the global edge-length ordering on the full mesh's adjacency? The answer changes whether `appliedFixes` is a true derivation log or a hybrid log+snapshot.

3. **What is the corpus-defined "50-face island"** that anchors the sub-second budget? Is it a specific piece in deer.obj? A synthetic worst-case? Without naming the benchmark target, the v3.5 spike has no objective pass/fail line.
